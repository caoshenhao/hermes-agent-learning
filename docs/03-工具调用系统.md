# 模块3：工具调用系统

工具调用系统是连接 LLM 推理与外部世界的桥梁。它负责：将 Python 函数转化为 JSON Schema 工具定义、将模型返回的工具调用分派到对应处理器、管理工具的生命周期与可用性。

## 一、核心文件概览

| 文件 | 职责 | 行数 |
|---|---|---|
| `tools/registry.py` | 工具注册中心（单例），存储所有工具的 schema、handler、元数据 | 589 |
| `model_tools.py` | 编排层：触发工具发现、提供 `get_tool_definitions()` 和 `handle_function_call()` | 1,216 |
| `toolsets.py` | 工具集定义，声明式地将工具按功能分组 | - |
| `agent/tool_executor.py` | 工具执行器：顺序/并发执行工具调用 | - |
| `tools/schema_sanitizer.py` | Schema 消毒器：修复不兼容的 JSON Schema 结构 | - |
| `tools/*.py`（82 个文件） | 具体工具实现，每个在模块级别调用 `registry.register()` | - |

## 二、工具注册机制（Auto-Discovery）

### 2.1 导入链（无循环导入）

```
tools/registry.py          （零依赖 — 所有工具文件导入它）
       ↑
tools/*.py                 （每个工具文件在 import 时调用 registry.register()）
       ↑
model_tools.py             （导入 tools.registry + 触发工具发现）
       ↑
run_agent.py, cli.py, batch_runner.py 等入口
```

### 2.2 自动发现流程

`model_tools.py` 在模块级别执行 `discover_builtin_tools()`：

```python
# model_tools.py
discover_builtin_tools()  # 模块加载时立即执行
```

`discover_builtin_tools()` 的工作步骤：

```python
def discover_builtin_tools(tools_dir=None):
    tools_path = Path(__file__).resolve().parent / "tools"
    # 1. 扫描 tools/ 目录下所有 .py 文件
    # 2. 用 AST 解析排除 __init__.py, registry.py, mcp_tool.py
    # 3. 用 AST 检测每个模块是否包含顶层 registry.register() 调用
    # 4. 对符合条件的模块调用 importlib.import_module()
    #    → 触发模块级代码 → 执行 registry.register()
```

**关键设计**：使用 AST 静态分析（而非正则）检测 `registry.register(...)` 调用。只检测模块体的顶层语句，忽略函数内部的调用。

### 2.3 注册数量

项目中有 **82 个** `tools/*.py` 文件，其中约 **40+ 个** 包含顶层 `registry.register()` 调用。部分文件注册多个工具：

| 工具文件 | 注册工具数 |
|---|---|
| `browser_tool.py` | 11 个浏览器子工具 |
| `kanban_tools.py` | 10 个看板工具 |
| `file_tools.py` | 4 个文件工具 |
| `web_tools.py` | 2 个（web_search, web_extract） |
| `feishu_drive_tool.py` | 4 个飞书文档工具 |

## 三、Registry 数据结构

### 3.1 ToolEntry

每个注册工具的元数据存储在 `ToolEntry` 中（使用 `__slots__` 优化内存）：

```python
class ToolEntry:
    __slots__ = (
        "name",                    # 工具名，如 "read_file"
        "toolset",                 # 所属工具集，如 "file"
        "schema",                  # JSON Schema dict（OpenAI function format）
        "handler",                 # Callable：实际的处理函数
        "check_fn",                # Optional[Callable]：可用性检查
        "requires_env",            # list[str]：所需环境变量
        "is_async",                # bool：是否异步处理函数
        "description",             # str：工具描述
        "emoji",                   # str：UI 显示 emoji
        "max_result_size_chars",   # int|float|None：输出截断阈值
        "dynamic_schema_overrides",# Optional[Callable]：运行时动态 schema 覆盖
    )
```

### 3.2 ToolRegistry（单例）

```python
class ToolRegistry:
    def __init__(self):
        self._tools: Dict[str, ToolEntry] = {}        # name → ToolEntry
        self._toolset_checks: Dict[str, Callable] = {} # toolset → check_fn
        self._toolset_aliases: Dict[str, str] = {}     # alias → canonical toolset
        self._lock = threading.RLock()                 # 线程安全
        self._generation: int = 0                      # 世代计数器（变更时递增）

# 模块级单例
registry = ToolRegistry()
```

**`_generation` 计数器**：每次 `register()` / `deregister()` 时递增。外部调用者可基于它判断缓存是否有效，实现 O(1) 缓存失效检测。

### 3.3 注册示例

```python
# tools/file_tools.py 末尾
registry.register(
    name="read_file",
    toolset="file",
    schema=READ_FILE_SCHEMA,           # 预定义的 JSON Schema dict
    handler=_handle_read_file,          # 处理函数
    check_fn=_check_file_reqs,          # 可用性检查
    emoji="📖",
    max_result_size_chars=100_000,      # 最大输出 100K 字符
)
```

## 五、工具定义管线（Tool Definition Pipeline）

### 5.1 完整流程

```
Python 函数 + JSON Schema dict（在 tools/*.py 中）
         │
         ▼
registry.register() → ToolEntry 存入 ToolRegistry._tools
         │
         ▼
get_tool_definitions(enabled_toolsets, disabled_toolsets)
    │
    ├── 1. resolve_toolset()：将 toolset 名解析为工具名列表
    ├── 2. 过滤 enabled/disabled toolsets → tools_to_include (Set[str])
    ├── 3. registry.get_definitions()：
    │       ├── check_fn 检查（TTL 缓存 30s）
    │       └── dynamic_schema_overrides 应用
    ├── 4. 动态 schema 重建（execute_code, discord）
    ├── 5. 跨引用清理（browser_navigate 引用 web_search 等）
    ├── 6. sanitize_tool_schemas()：修复不兼容 schema
    └── 7. Tool Search 组装（渐进式披露）
         │
         ▼
OpenAI-format tools array:
[{"type": "function", "function": {"name": "...", "parameters": {...}}}, ...]
```

### 5.2 缓存机制

- 仅 `quiet_mode=True` 时启用缓存
- 缓存键：`(frozenset(enabled), frozenset(disabled), registry._generation, config_mtime, ...)`
- `registry._generation` 变更自动使缓存失效
- 配置文件 mtime/size 变更也触发失效

### 5.3 过滤逻辑

```python
if enabled_toolsets is not None:
    for ts in enabled_toolsets:
        tools_to_include.update(resolve_toolset(ts))  # 并集
else:
    tools_to_include = 所有工具

if disabled_toolsets:
    for ts in disabled_toolsets:
        tools_to_include.difference_update(resolve_toolset(ts))  # 差集
```

## 六、工具调用分派流程（Tool Call Dispatch）

### 6.1 完整调用链

```
API 响应包含 tool_calls → run_agent.py 循环
         │
         ▼
model_tools.handle_function_call(name, args, task_id, ...)
         │
         ├── 1. coerce_tool_args()：类型强制转换
         ├── 2. Tool Search bridge 检查
         ├── 3. tool_request_middleware：请求中间件
         ├── 4. _AGENT_LOOP_TOOLS 检查（todo/memory 等）
         ├── 5. pre_tool_call hook：插件阻塞检查
         ├── 6. ACP edit approval：编辑审批
         ├── 7. 读取计数器更新
         ├── 8. registry.dispatch(name, args)
         │       ├── 查找 ToolEntry
         │       ├── 异步 → _run_async(handler(args))
         │       └── 同步 → handler(args)
         ├── 9. post_tool_call hook：后置观察钩子
         ├── 10. transform_tool_result hook：结果变换
         └── 返回 JSON 字符串结果
```

### 6.2 registry.dispatch()

```python
def dispatch(self, name, args, **kwargs) -> str:
    entry = self.get_entry(name)
    if not entry:
        return json.dumps({"error": f"Unknown tool: {name}"})
    if entry.is_async:
        return _run_async(entry.handler(args, **kwargs))
    return entry.handler(args, **kwargs)
```

### 6.3 异步桥接（_run_async）

同步→异步的唯一桥梁，处理三种场景：

| 场景 | 策略 |
|---|---|
| CLI 主线程（无运行事件循环） | 使用持久事件循环 `_tool_loop` |
| 工作线程（如 delegate_task 并行执行） | 每线程持久循环 `_worker_thread_local` |
| 异步上下文内（如 gateway） | 在新线程中创建临时事件循环，300s 超时 |

### 6.4 参数类型强制转换

LLM 经常返回错误类型的参数。`coerce_tool_args()` 处理：

```
"42"       → 42          （字符串→整数）
"true"     → true        （字符串→布尔）
"null"     → None        （字符串→None）
"https://a.com" → ["https://a.com"]  （裸值→数组包装）
'{"a":1}'  → {"a": 1}   （JSON字符串→对象）
```

## 七、Schema 消毒管线

### 7.1 问题背景

某些本地推理后端（如 llama.cpp）对 JSON Schema 非常严格。OpenAI 默默接受的 schema 会让它们崩溃。

### 7.2 处理的不兼容结构

| 问题 | 修复方式 |
|---|---|
| `{"type": "object"}` 无 `properties` | 注入 `"properties": {}` |
| schema 值是裸字符串 `"object"` | 替换为完整 dict |
| `"type": ["string", "null"]` 数组类型 | 折叠为单一类型 |
| `anyOf`/`oneOf` 仅用于允许 null | 折叠到非 null 分支 |
| 顶层 `allOf`/`anyOf`/`oneOf`/`enum` | 剥离 |

所有修改在**深拷贝**上进行，不影响注册表中的原始 schema。

## 九、check_fn 缓存机制

`check_fn` 用于探测外部状态（Docker daemon、Modal SDK 等），调用代价高。Registry 使用 **30 秒 TTL 缓存**：

```python
_CHECK_FN_TTL_SECONDS = 30.0
_check_fn_cache: Dict[Callable, tuple[float, bool]] = {}

def _check_fn_cached(fn: Callable) -> bool:
    """返回 bool(fn())，TTL 缓存，异常视为 False"""
```

显式失效：`invalidate_check_fn_cache()` 在 `hermes tools enable` 等配置变更后调用。

## 十一、Tool Search（渐进式工具披露）

当 MCP + 插件工具占用超过上下文窗口的 10% 时，系统自动将这些非核心工具替换为 3 个桥接工具：

| 桥接工具 | 功能 |
|---|---|
| `tool_search` | 搜索可用工具 |
| `tool_describe` | 查看工具详细 schema |
| `tool_call` | 调用被延迟的工具 |

核心工具（`_HERMES_CORE_TOOLS` 中的工具）**永不延迟**。

## 十二、工具执行并发模型

`agent/tool_executor.py` 管理两种执行模式：

| 模式 | 方法 | 说明 |
|---|---|---|
| 顺序执行 | `_execute_tool_calls_sequential()` | 逐个执行 |
| 并发执行 | `_execute_tool_calls_concurrent()` | `ThreadPoolExecutor(max_workers=8)` |

每个工具调用经过：

```
API 返回 tool_calls
    → tool guardrails（破坏性操作检查）
    → middleware（请求修改）
    → registry.dispatch()
    → 结果存储（maybe_persist_tool_result）
    → 预算追踪（enforce_turn_budget）
    → 中断检查（_set_interrupt）
```

## 十三、如何添加新工具（Step by Step）

### 步骤 1：创建工具文件

在 `tools/` 目录下创建 `my_tool.py`：

```python
"""My custom tool for Hermes Agent."""
import json
from tools.registry import registry

MY_TOOL_SCHEMA = {
    "name": "my_tool",
    "description": "Does something useful",
    "parameters": {
        "type": "object",
        "properties": {
            "input": {
                "type": "string",
                "description": "The input to process"
            }
        },
        "required": ["input"]
    }
}

def _check_my_tool():
    """检查依赖是否可用"""
    try:
        import some_dependency
        return True
    except ImportError:
        return False

def _handle_my_tool(args, **kwargs):
    input_text = args.get("input", "")
    return json.dumps({"result": f"Processed: {input_text}"})

# 模块级注册（必须有！）
registry.register(
    name="my_tool",
    toolset="my_toolset",
    schema=MY_TOOL_SCHEMA,
    handler=_handle_my_tool,
    check_fn=_check_my_tool,
    emoji="🎯",
    max_result_size_chars=50_000,
)
```

```python
TOOLSETS["my_toolset"] = {

    "tools": ["my_tool"],
    "includes": [],
}
```

## 十四、关键设计决策总结

| 设计决策 | 原因 |
|---|---|
| AST 检测 + importlib | 安全地自动发现工具，无需手动维护导入列表 |
| `__slots__` ToolEntry | 82+ 工具时节省内存 |
| `threading.RLock` + 快照 | MCP 动态刷新可在多线程环境中安全变更 |
| `_generation` 计数器 | O(1) 缓存失效检测 |
| check_fn 30s TTL | 平衡性能和响应性 |
| `override=False` 保护 | 防止插件/MCP 意外覆盖内置工具 |
| Schema 消毒 | 兼容 llama.cpp 等严格后端 |
| 持久事件循环 | 避免异步工具的 "Event loop is closed" 错误 |
| Tool Search 桥接 | 大量 MCP/插件工具时不浪费上下文窗口 |
