# 模块3：工具调用系统

工具调用系统是连接 LLM 推理与外部世界的桥梁。它负责：将 Python 函数转化为 JSON Schema 工具定义、将模型返回的工具调用分派到对应处理器、管理工具的生命周期与可用性。

## 一、核心文件概览

| 文件 | 职责 | 行数 |
|---|---|---|
| `tools/registry.py` | 工具注册中心（单例），存储所有工具的 schema、handler、元数据 | 589 |
| `model_tools.py` | 编排层：触发工具发现、提供 `get_tool_definitions()` 和 `handle_function_call()` | 1,216 |
| `toolsets.py` | 工具集定义，声明式地将工具按功能分组 | - |
| `agent/tool_executor.py` | 工具执行器：顺序/并发执行工具调用 | - |
| `tools/schema_sanitizer.py` | Schema 消毒器：修复不兼容的 JSON Schema 结构 | - |
| `tools/*.py`（82 个文件） | 具体工具实现，每个在模块级别调用 `registry.register()` | - |

## 二、工具注册机制（Auto-Discovery）

### 2.1 导入链（无循环导入）

```
tools/registry.py          （零依赖 — 所有工具文件导入它）
       ↑
tools/*.py                 （每个工具文件在 import 时调用 registry.register()）
       ↑
model_tools.py             （导入 tools.registry + 触发工具发现）
       ↑
run_agent.py, cli.py, batch_runner.py 等入口
```

### 2.2 自动发现流程

`model_tools.py` 在模块级别执行 `discover_builtin_tools()`：

```python
# model_tools.py
discover_builtin_tools()  # 模块加载时立即执行
```

`discover_builtin_tools()` 的工作步骤：

```python
def discover_builtin_tools(tools_dir=None):
    tools_path = Path(__file__).resolve().parent / "tools"
    # 1. 扫描 tools/ 目录下所有 .py 文件
    # 2. 用 AST 解析排除 __init__.py, registry.py, mcp_tool.py
    # 3. 用 AST 检测每个模块是否包含顶层 registry.register() 调用
    # 4. 对符合条件的模块调用 importlib.import_module()
    #    → 触发模块级代码 → 执行 registry.register()
```

**关键设计**：使用 AST 静态分析（而非正则）检测 `registry.register(...)` 调用。只检测模块体的顶层语句，忽略函数内部的调用。

### 2.3 注册数量

项目中有 **82 个** `tools/*.py` 文件，其中约 **40+ 个** 包含顶层 `registry.register()` 调用。部分文件注册多个工具：

| 工具文件 | 注册工具数 |
|---|---|
| `browser_tool.py` | 11 个浏览器子工具 |
| `kanban_tools.py` | 10 个看板工具 |
| `file_tools.py` | 4 个文件工具 |
| `web_tools.py` | 2 个（web_search, web_extract） |
| `feishu_drive_tool.py` | 4 个飞书文档工具 |

## 三、Registry 数据结构

### 3.1 ToolEntry

每个注册工具的元数据存储在 `ToolEntry` 中（使用 `__slots__` 优化内存）：

```python
class ToolEntry:
    __slots__ = (
        "name",                    # 工具名，如 "read_file"
        "toolset",                 # 所属工具集，如 "file"
        "schema",                  # JSON Schema dict（OpenAI function format）
        "handler",                 # Callable：实际的处理函数
        "check_fn",                # Optional[Callable]：可用性检查
        "requires_env",            # list[str]：所需环境变量
        "is_async",                # bool：是否异步处理函数
        "description",             # str：工具描述
        "emoji",                   # str：UI 显示 emoji
        "max_result_size_chars",   # int|float|None：输出截断阈值
        "dynamic_schema_overrides",# Optional[Callable]：运行时动态 schema 覆盖
    )
```

### 3.2 ToolRegistry（单例）

```python
class ToolRegistry:
    def __init__(self):
        self._tools: Dict[str, ToolEntry] = {}        # name → ToolEntry
        self._toolset_checks: Dict[str, Callable] = {} # toolset → check_fn
        self._toolset_aliases: Dict[str, str] = {}     # alias → canonical toolset
        self._lock = threading.RLock()                 # 线程安全
        self._generation: int = 0                      # 世代计数器（变更时递增）

# 模块级单例
registry = ToolRegistry()
```

**`_generation` 计数器**：每次 `register()` / `deregister()` 时递增。外部调用者可基于它判断缓存是否有效，实现 O(1) 缓存失效检测。

### 3.3 注册示例

```python
# tools/file_tools.py 末尾
registry.register(
    name="read_file",
    toolset="file",
    schema=READ_FILE_SCHEMA,           # 预定义的 JSON Schema dict
    handler=_handle_read_file,          # 处理函数
    check_fn=_check_file_reqs,          # 可用性检查
    emoji="📖",
    max_result_size_chars=100_000,      # 最大输出 100K 字符
)
```

## 五、工具定义管线（Tool Definition Pipeline）

### 5.1 完整流程

```
Python 函数 + JSON Schema dict（在 tools/*.py 中）
         │
         ▼
registry.register() → ToolEntry 存入 ToolRegistry._tools
         │
         ▼
get_tool_definitions(enabled_toolsets, disabled_toolsets)
    │
    ├── 1. resolve_toolset()：将 toolset 名解析为工具名列表
    ├── 2. 过滤 enabled/disabled toolsets → tools_to_include (Set[str])
    ├── 3. registry.get_definitions()：
    │       ├── check_fn 检查（TTL 缓存 30s）
    │       └── dynamic_schema_overrides 应用
    ├── 4. 动态 schema 重建（execute_code, discord）
    ├── 5. 跨引用清理（browser_navigate 引用 web_search 等）
    ├── 6. sanitize_tool_schemas()：修复不兼容 schema
    └── 7. Tool Search 组装（渐进式披露）
         │
         ▼
OpenAI-format tools array:
[{"type": "function", "function": {"name": "...", "parameters": {...}}}, ...]
```

### 5.2 缓存机制

- 仅 `quiet_mode=True` 时启用缓存
- 缓存键：`(frozenset(enabled), frozenset(disabled), registry._generation, config_mtime, ...)`
- `registry._generation` 变更自动使缓存失效
- 配置文件 mtime/size 变更也触发失效

### 5.3 过滤逻辑

```python
if enabled_toolsets is not None:
    for ts in enabled_toolsets:
        tools_to_include.update(resolve_toolset(ts))  # 并集
else:
    tools_to_include = 所有工具

if disabled_toolsets:
    for ts in disabled_toolsets:
        tools_to_include.difference_update(resolve_toolset(ts))  # 差集
```

## 六、工具调用分派流程（Tool Call Dispatch）

### 6.0 完整管线概览

工具调用从 API 响应到结果回传，经过 **四个阶段、十余个步骤**。以下按实际代码执行顺序描述。

```

    ├── 0a. _deduplicate_tool_calls()：去除重复 (name, args) 对
    ├── 0b. _cap_delegate_task_calls()：截断超出 max_concurrent_children 的 delegate_task
    ├── 0c. _should_parallelize_tool_batch()：决定顺序 vs 并发执行
    │
    ▼ 对每个 tool_call 依次（顺序）或并行（ThreadPoolExecutor）处理：

━━━ 阶段2：单工具执行前置检查（tool_executor.py preflight）━━━

    ├── 1. Tool Search unwrap（如果 tool_name == "tool_call"）
    │       → 解包为底层工具名+参数，让下游所有 hook 看到真实工具
    │       → 会话范围检查：底层工具必须在 session 的 deferrable 集合中
    │
    ├── 2. tool_request_middleware：请求中间件（可修改参数）
    │
    ├── 3. 三层 Block 检查（BEFORE checkpoint）：
    │       3a. Tool Search scope block（越权工具调用）
    │       3b. pre_tool_call plugin hook（插件阻塞）

    │
    ├── 4. Checkpoint preflight（仅对将执行的工具有效）：
    │       4a. write_file / patch → 对目标路径做 git checkpoint
    │       4b. terminal + _is_destructive_command → 对 cwd 做 checkpoint

━━━ 阶段3：实际工具执行（model_tools.handle_function_call）━━━
```

### 6.1 registry.dispatch()

```python
def dispatch(self, name, args, **kwargs) -> str:
    entry = self.get_entry(name)
    if not entry:
        return json.dumps({"error": f"Unknown tool: {name}"})
    if entry.is_async:
        return _run_async(entry.handler(args, **kwargs))
    return entry.handler(args, **kwargs)
```

### 6.2 异步桥接（_run_async）

同步→异步的唯一桥梁，处理三种场景：

| 场景 | 策略 |
|---|---|
| CLI 主线程（无运行事件循环） | 使用持久事件循环 `_tool_loop` |
| 工作线程（如 delegate_task 并行执行） | 每线程持久循环 `_worker_thread_local` |
| 异步上下文内（如 gateway） | 在新线程中创建临时事件循环，300s 超时 |

### 6.3 参数类型强制转换（coerce_tool_args）

LLM 经常返回错误类型的参数。`coerce_tool_args()` 处理：

```
"42"       → 42          （字符串→整数）
"true"     → true        （字符串→布尔）
"null"     → None        （字符串→None）
"https://a.com" → ["https://a.com"]  （裸值→数组包装）
'{"a":1}'  → {"a": 1}   （JSON字符串→对象）
```

实现细节：通过 `registry.get_schema(tool_name)` 获取工具的 JSON Schema，逐参数比对 `properties[key].type`，仅对字符串值尝试强制转换。数组类型有特殊处理：先尝试 `_coerce_value` 解析 JSON 字符串，失败则包装为单元素列表。

### 6.4 Agent 级工具（invoke_tool）

并非所有工具都经过 `registry.dispatch()`。`agent/agent_runtime_helpers.py:invoke_tool()` 是并发执行路径的入口，它先拦截 **Agent 级工具**（需要访问 Agent 实例状态的工具），再将其余工具委托给 `handle_function_call()`：

| Agent 级工具 | 为什么需要 Agent 实例 |
|---|---|
| `todo` | 访问 `agent._todo_store` |
| `session_search` | 访问 `agent._get_session_db_for_recall()` |
| `memory` | 访问 `agent._memory_store` + 通知 `agent._memory_manager` |
| `clarify` | 访问 `agent.clarify_callback` |
| `delegate_task` | 访问 `agent._dispatch_delegate_task()` |

`_AGENT_LOOP_TOOLS = {"todo", "memory", "session_search", "delegate_task"}` 集合定义在 `model_tools.py` 中。当 `handle_function_call()` 遇到这些工具时返回错误——它们必须由 Agent Loop 层处理。

### 6.5 Tool Execution Middleware

实际的工具执行被 `run_tool_execution_middleware()` 包裹。这是一个可插拔的中间件层，插件可以在工具实际执行前后插入逻辑（类似 Web 框架的请求中间件）。在 `invoke_tool()` 中，`_execute` 闭包先经过 `run_tool_execution_middleware`，然后才调用 `registry.dispatch()`。

## 七、Schema 消毒管线

### 7.1 问题背景

某些本地推理后端（如 llama.cpp）对 JSON Schema 非常严格。OpenAI 默默接受的 schema 会让它们崩溃。

### 7.2 处理的不兼容结构

| 问题 | 修复方式 |
|---|---|
| `{"type": "object"}` 无 `properties` | 注入 `"properties": {}` |
| schema 值是裸字符串 `"object"` | 替换为完整 dict |
| `"type": ["string", "null"]` 数组类型 | 折叠为单一类型 |
| `anyOf`/`oneOf` 仅用于允许 null | 折叠到非 null 分支 |
| 顶层 `allOf`/`anyOf`/`oneOf`/`enum` | 剥离 |

所有修改在**深拷贝**上进行，不影响注册表中的原始 schema。

## 七B、工具名修复（Tool Name Repair）

源码：`agent/agent_runtime_helpers.py:repair_tool_call()`

模型有时会输出工具名的变体——大小写不同、带分隔符不同、加了类名后缀。在返回 "Unknown tool" 之前，Hermes 会尝试 5 步修复：

| 步骤 | 策略 | 示例 |
|---|---|---|
| 1 | 小写直接匹配 | `ReadFile` → `read_file` |
| 2 | 连字符/空格 → 下划线 + 小写 | `read-file` → `read_file` |
| 3 | CamelCase → snake_case | `TodoTool` → `todo_tool` |
| 4 | 剥离 `_tool` / `-tool` / `tool` 后缀（递归两次） | `TodoTool_tool` → `TodoTool` → `todo_tool` → `todo` |
| 5 | 模糊匹配（difflib, cutoff=0.7） | `browser_clic` → `browser_click` |

步骤 4 的递归设计是因为 Claude 式模型有时双重叠加后缀（如 `TodoTool_tool`），单次剥离不够。该函数返回修复后的工具名（如果能在 `valid_tool_names` 中找到）或 `None`。

## 七C、工具调用预处理

源码：`run_agent.py`（`_deduplicate_tool_calls` / `_cap_delegate_task_calls`）

在工具调用进入执行管线之前，有两个预处理步骤：

### 去重（_deduplicate_tool_calls）

在同一轮 API 响应中，如果模型发出了多个完全相同的 `(tool_name, arguments)` 对，只保留第一个。重复的调用会被静默丢弃并记录 warning。

### Delegate 截断（_cap_delegate_task_calls）

`delegate_task` 工具在单次调用内部已经限制了任务数，但模型可能在一轮中发出多个独立的 `delegate_task` 调用。此函数截断超出 `max_concurrent_children` 的调用，保留所有非 delegate 调用。

## 七D、工具调用 Guardrails（循环检测）

源码：`agent/tool_guardrails.py`

`ToolCallGuardrailController` 是一个 **每轮重置** 的控制器，检测三种工具调用循环模式：

| 检测类型 | 描述 | 默认 warn | 默认 block/halt |
|---|---|---|---|
| `repeated_exact_failure` | 同一 (tool, args) 组合连续失败 | 2 次 | 5 次 block |
| `same_tool_failure` | 同一工具名（不同参数）连续失败 | 3 次 | 8 次 halt |
| `idempotent_no_progress` | 幂等工具（read_file 等）返回相同结果 | 2 次 | 5 次 block |

### 关键数据结构

- **ToolCallSignature**：`(tool_name, sha256(canonical_args))` — 不可逆的身份标识
- **canonical_tool_args**：对参数做 `sort_keys=True` 的 JSON 序列化，确保参数顺序不影响签名

### 两阶段检查

```python
# before_call：在工具执行前检查历史计数
decision = controller.before_call(tool_name, args)
if not decision.allows_execution:
    # block 或 halt → 返回错误，不执行工具

# after_call：在工具执行后更新计数
decision = controller.after_call(tool_name, args, result, failed=...)
if decision.action == "warn":
    # 将 warning 文本追加到工具结果
```

### 分类常量

| 集合 | 包含工具 |
|---|---|
| `IDEMPOTENT_TOOL_NAMES` | read_file, search_files, web_search, web_extract, session_search, browser_snapshot, browser_console 等（15 个） |
| `MUTATING_TOOL_NAMES` | terminal, execute_code, write_file, patch, todo, memory, skill_manage, browser_click/type/press/scroll/navigate, send_message, cronjob, delegate_task, process（15 个） |

### 配置（config.yaml）

```yaml
tool_loop_guardrails:
  warnings_enabled: true      # 默认开
  hard_stop_enabled: false    # 默认关 — CLI/TUI 不强制阻断
  warn_after:
    exact_failure: 2
    same_tool_failure: 3
    idempotent_no_progress: 2
  hard_stop_after:
    exact_failure: 5
    same_tool_failure: 8
    idempotent_no_progress: 5
```

## 七E、Strict API 消毒

源码：`run_agent.py`（`_sanitize_tool_calls_for_strict_api` / `_should_sanitize_tool_calls`）

Codex Responses API 使用非标准字段（`call_id`、`response_item_id`、`extra_content`），这些字段在标准 Chat Completions API 中会被 Mistral、Fireworks 等严格 Provider 拒绝（400/422）。

### 消毒逻辑

- `_should_sanitize_tool_calls()`：当 `api_mode != "codex_responses"` 时返回 True
- `_sanitize_tool_calls_for_strict_api()`：创建新的 tool_call dict（不原地修改），剥离 `call_id` 和 `response_item_id`
- `extra_content`（Gemini `thought_signature`）：仅在模型不是 Gemini 系列时剥离（Gemini 3 thinking 模型无此字段会 400）

消息历史保留完整字段（以支持后续 fallback 到 Codex provider），只修改发给 API 的副本。

## 十一、Tool Search（渐进式工具披露）

源码：`tools/tool_search.py`（735 行）

### 11.1 设计目标

当 MCP + 插件工具的 Schema 占用超过上下文窗口的 10% 时，系统自动将这些非核心工具替换为 3 个桥接工具，按需加载：

| 桥接工具 | 功能 |
|---|---|
| `tool_search` | 搜索可用工具（关键词匹配，返回工具名+简短描述） |
| `tool_describe` | 查看单个工具的完整 JSON Schema |
| `tool_call` | 调用被延迟的工具（参数必须与 schema 匹配） |

核心工具（`_HERMES_CORE_TOOLS` 中的工具）**永不延迟**。

### 11.2 激活门控（should_activate）

```python
def should_activate(config, deferrable_tokens, context_length) -> bool:
    if config.enabled == "off": return False          # 显式关闭
    if deferrable_tokens <= 0: return False            # 无可延迟工具
    if config.enabled == "on": return True             # 强制开启
    # auto 模式
    if not context_length:
        return deferrable_tokens >= 20_000             # 未知上下文大小 → 固定 20K token 阈值
    threshold_tokens = int(context_length * config.threshold_pct / 100)
    return deferrable_tokens >= threshold_tokens       # 默认 10% of context
```

- **Token 估算**：`chars / 4` 规则（`CHARS_PER_TOKEN=4`），无需精确 tokenizer
- **auto 模式**：默认行为。200K context × 10% = 20K token 阈值
- **on/off 模式**：配置强制启用或禁用

### 11.3 无状态目录

Tool Search 的目录 **不做会话级缓存**，每次组装时从当前 `get_tool_definitions()` 重建。这是从 OpenClaw cron 回归（#84141）中吸取的教训：会话级缓存的目录与实时注册表漂移时会导致静默的工具丢失。

### 11.4 会话范围强制

桥接工具通过 `model_tools.handle_function_call()` 分发，但有两道范围防线确保受限会话（子 Agent、看板 Worker、网关会话）不会通过桥接调用未授权的工具：

**防线 1（tool_executor.py preflight）**：在工具执行前，`_tool_search_scoped_names(agent)` 返回当前会话可延迟的工具名集合（基于 `enabled_toolsets` / `disabled_toolsets`）。如果底层工具不在集合中，直接返回错误。

**防线 2（handle_function_call bridge dispatch）**：当 `tool_call` 桥接被调用时，递归调用 `handle_function_call` 传递底层工具名之前，再次检查 `scoped_deferrable_names(current_defs)`。

```python
# model_tools.py — tool_call 桥接分发
underlying_name, underlying_args, err = resolve_underlying_call(function_args)
_scoped_deferrable = scoped_deferrable_names(current_defs)
if underlying_name not in _scoped_deferrable:
    return {"error": f"'{underlying_name}' is not available in this session."}
# 递归 — 所有 hook 对底层工具名触发
return handle_function_call(underlying_name, underlying_args, ...)
```

### 11.5 桥接 unwrap 设计

当模型通过 `tool_call` 桥接调用工具时，`tool_executor.py` 会在 preflight 阶段将 `tool_call` **unwrap** 为底层工具名+参数：

- **原 tool_call 条目保留不变**（保持对话记录和 tool_call_id 的完整性）
- **所有下游检查（checkpoint、guardrail、plugin hook、display）看到真实工具名**
- 这确保了 OpenClaw 教训成立：hook 必须观察到真实工具名，而非桥接名

### 11.6 显示和轨迹展开

`tool_search.py` 包含 display/trajectory unwrap 逻辑，确保用户（CLI 活动流、网关、保存的轨迹文件）始终看到底层工具，而非桥接工具。

## 十二、工具执行并发模型

源码：`agent/tool_executor.py`（1409 行）、`agent/tool_dispatch_helpers.py`

### 12.1 并行决策（_should_parallelize_tool_batch）

不是所有工具批次都能安全并发。决策逻辑分三类：

| 分类 | 集合 | 说明 |
|---|---|---|
| **永不并行** | `_NEVER_PARALLEL_TOOLS` = `{clarify}` | 出现就强制顺序执行 |
| **并行安全** | `_PARALLEL_SAFE_TOOLS` | 只读、无共享可变状态的工具（read_file, search_files, session_search, skill_view, skills_list, vision_analyze, web_extract, web_search, ha_*） |
| **路径隔离** | `_PATH_SCOPED_TOOLS` = `{read_file, write_file, patch}` | 按文件路径隔离 — 路径不重叠时可并行 |

额外规则：
- MCP 工具可通过服务器配置 opt-in 到并行（`_is_mcp_tool_parallel_safe`）
- 路径重叠检测用 `_paths_overlap()` — 规范化后比较路径前缀
- 无法解析参数的调用 → 回退顺序执行（保守策略）

### 12.2 两种执行模式

| 模式 | 函数 | 使用场景 |
|---|---|---|
| 顺序执行 | `execute_tool_calls_sequential()` | 单工具调用、`clarify` 在批次中、或并行不安全 |
| 并发执行 | `execute_tool_calls_concurrent()` | 并行安全的一批工具 |

### 12.3 并发执行管线（完整流程）

```
execute_tool_calls_concurrent() 内部：

1. 中断检查 → 如果已中断，跳过所有工具调用
2. 逐个解析参数 + preflight 检查：
   a. Tool Search unwrap（解包 tool_call 桥接）
   b. tool_request_middleware（可修改参数）
   c. 三层 Block 检查（scope / plugin / guardrail）
   d. Checkpoint preflight（write_file/patch/destructive terminal）
3. 启动 ThreadPoolExecutor(max_workers=min(N, 8))
4. 每个工作线程：
   a. 注册线程 ID（用于中断传播）
   b. 传播 ContextVar + 审批回调到工作线程
   c. 调用 agent._invoke_tool()
   d. 异常/中断处理
   e. finally: 注销线程 ID + 清除中断标记
5. 主线程轮询等待（5s 间隔）：
   a. 每 30s 发送心跳（网关活跃监控）
   b. 检测中断 → cancel 未开始的 future + 等待 3s 让运行中的优雅退出
6. 收集结果 → 按原始顺序排列
7. 逐个结果后处理（guardrail after_call, persist, budget, display）
```

### 12.4 线程中断传播

并发执行时，用户中断需要传播到所有工作线程：

```python
# 工作线程注册
_worker_tid = threading.current_thread().ident
agent._tool_worker_threads.add(_worker_tid)

# 主线程收到中断
agent.interrupt() → 遍历 _tool_worker_threads，对每个 tid 设置中断标记

# 工作线程内
if agent._interrupt_requested:
    _ra()._set_interrupt(True, _worker_tid)  # 对自己应用中断

# finally 清理
agent._tool_worker_threads.discard(_worker_tid)
_ra()._set_interrupt(False, _worker_tid)  # 清除标记，防止 tid 回收后中毒
```

### 12.5 破坏性命令检测

`_is_destructive_command(cmd)` 用正则启发式判断终端命令是否可能修改/删除文件：

匹配的模式：`rm`, `rmdir`, `cp`, `mv`, `install`, `sed -i`, `truncate`, `dd`, `shred`, `git reset/clean/checkout`，以及文件覆盖重定向 `>` (非 `>>`)。

检测到破坏性命令时，在执行前对 cwd 做 git checkpoint（如果 checkpoint 功能已启用）。

### 12.6 结果后处理

每个工具结果在追加到 messages 之前，经过：

| 步骤 | 函数 | 作用 |
|---|---|---|
| Guardrail 观察 | `after_call()` | 更新循环检测计数 |
| 文件变更追踪 | `_record_file_mutation_result()` | 供轮末验证器使用 |
| 大结果持久化 | `maybe_persist_tool_result()` | 超阈值的结果存磁盘，messages 中放引用 |
| Token 预算 | `enforce_turn_budget()` | 截断超长结果 |
| 子目录提示 | `_subdirectory_hints.check_tool_call()` | 追加路径建议 |
| 多模态适配 | `_tool_result_content_for_active_model()` | 转 vision-capable 格式 |

## 十四、关键设计决策总结

| 设计决策 | 原因 |
|---|---|
| AST 检测 + importlib | 安全地自动发现工具，无需手动维护导入列表 |
| `__slots__` ToolEntry | 82+ 工具时节省内存 |
| `threading.RLock` + 快照 | MCP 动态刷新可在多线程环境中安全变更 |
| `_generation` 计数器 | O(1) 缓存失效检测 |
| check_fn 30s TTL | 平衡性能和响应性 |
| `override=False` 保护 | 防止插件/MCP 意外覆盖内置工具 |
| Schema 消毒 | 兼容 llama.cpp 等严格后端 |
| 持久事件循环 | 避免异步工具的 "Event loop is closed" 错误 |
| Tool Search 桥接 | 大量 MCP/插件工具时不浪费上下文窗口 |
| 5 步工具名修复 | 模型常输出变体名（TodoTool_tool 等），修复比报错更高效 |
| 每轮重置 Guardrail | 循环检测只关注当前轮，避免历史误判 |
| ToolCallSignature 哈希签名 | 不可逆身份标识保护参数隐私 |
| Guardrail 默认 warn 不 block | 交互式 CLI/TUI 优先灵活性，hard_stop 需显式配置 |
| Strict API 消毒 | Codex Responses API 的非标字段会让 Mistral/Fireworks 400 |
| Tool Search unwrap 到底层工具 | 所有 hook/checkpoint/display 看到真实工具名（OpenClaw 教训） |
| 两道会话范围防线 | 受限会话不能通过桥接绕过工具集限制 |
| 并行安全的路径隔离 | write_file/patch/read_file 路径不重叠时可安全并发 |
| ContextVar 传播到工作线程 | 审批回调/observability 上下文不丢（#13617 安全修复） |
| 心跳 30s 间隔 | 并发工具执行时保持网关活跃监控 |
